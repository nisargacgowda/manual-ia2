What is JUnit? And its features

JUnit is a widely used Java testing framework that helps developers write and run unit tests for their applications. It is mainly used in test-driven development (TDD) to ensure that individual components (methods/classes) of a program work correctly.

In simple terms:
👉 JUnit helps you automatically test your Java code instead of checking everything manually.

✨ Features of JUnit

Here are the key features you should remember (perfect for exam answers 👇):

1. Annotations

JUnit uses annotations to define test methods and control execution.

Examples:

@Test → marks a method as a test case
@BeforeEach → runs before each test
@AfterEach → runs after each test

👉 Makes code clean and easy to understand.

2. Assertions

Used to check whether the expected output matches the actual result.

Examples:

assertEquals(expected, actual)
assertTrue(condition)
assertFalse(condition)

👉 Helps verify correctness automatically.

3. Test Suites

Allows grouping multiple test cases into a single suite.

👉 Useful for running multiple tests together.

4. Test Runners

JUnit provides test runners to execute test cases and display results.

👉 Shows:

Passed tests ✅
Failed tests ❌
5. Fixtures (Setup & Teardown)

Provides methods to prepare test environment before execution and clean up after.

Examples:

@BeforeEach
@AfterEach

👉 Ensures consistent testing conditions.

6. Parameterized Tests

Allows running the same test with different inputs.

👉 Saves time and improves coverage.

7. Integration with Tools

JUnit easily integrates with:

Maven
Gradle
IDEs like Eclipse, IntelliJ

👉 Makes automation easier.

8. Exception Testing

JUnit can test whether expected exceptions are thrown.

👉 Ensures error handling works correctly.

what are the important annotations provided by JUnit? 
Important Annotations in JUnit

JUnit provides annotations to control the execution of test cases and setup/cleanup processes.

1. @Test
Marks a method as a test method.
JUnit will execute this method during testing.
2. @BeforeEach
Executed before each test method.
Used for initialization (e.g., creating objects, setting values).
3. @AfterEach
Executed after each test method.
Used for cleanup activities.
4. @BeforeAll
Executed once before all test methods in the class.
Used for one-time setup.
5. @AfterAll
Executed once after all test methods.
Used for final cleanup.
6. @Disabled
Used to skip a test method.
Helpful when a test is not ready or temporarily not needed.
7. @DisplayName
Provides a custom name for test classes or methods.
Makes test reports more readable.
8. @ParameterizedTest
Used to run a test method with different input values.
9. @Nested
Allows grouping related test cases inside a class.
10. @Tag
Used to categorize tests (e.g., fast, slow).
Helps in selective execution.

what are the JUnit assert methods? 
JUnit Assert Methods
Assertions in JUnit are used to verify expected results with actual results. If the condition fails, the test case fails.
Common Assert Methods
1. assertEquals(expected, actual)
Checks if two values are equal.
2. assertNotEquals(expected, actual)
Checks if two values are not equal.
3. assertTrue(condition)
Passes if the condition is true.
4. assertFalse(condition)
Passes if the condition is false.
5. assertNull(object)
Checks if the object is null.
6. assertNotNull(object)
Checks if the object is not null.
7. assertSame(expected, actual)
Checks if both references point to the same object.
8. assertNotSame(expected, actual)
Checks if references point to different objects.
9. assertArrayEquals(expectedArray, actualArray)
Checks if two arrays are equal.
10. assertThrows(Exception.class, executable)
Verifies that a specific exception is thrown.
11. fail()
Explicitly fails a test case.

All these methods are available in:
org.junit.jupiter.api.Assertions

how do you test exceptions in JUnit 5?
Testing Exceptions in JUnit 5

In JUnit 5, exceptions are tested using the assertThrows() method from org.junit.jupiter.api.Assertions.

1. Using assertThrows()
Verifies that a specific exception is thrown.
Returns the exception object (can be used for further checks).

Syntax:

assertThrows(ExceptionType.class, () -> {
    // code that should throw exception
});
2. Example
import static org.junit.jupiter.api.Assertions.*;
import org.junit.jupiter.api.Test;

class TestExample {

    @Test
    void testException() {
        assertThrows(ArithmeticException.class, () -> {
            int result = 10 / 0;
        });
    }
}
3. Checking Exception Message

You can also verify the exception message:

@Test
void testExceptionMessage() {
    Exception ex = assertThrows(ArithmeticException.class, () -> {
        int result = 10 / 0;
    });

    assertEquals("/ by zero", ex.getMessage());
}
4. Key Points
Uses lambda expression to execute code.
Test passes only if the expected exception is thrown.
If no exception or a different exception occurs → test fails.

Easy to visualize coverage.

Maven

Can enforce minimum coverage rules.
7. Lightweight and Fast
Adds minimal overhead during test execution.
Easy to configure and use.

how do you generate and run a JaCoCo test report? 
Generating and Running a JaCoCo Test Report

JaCoCo is commonly used with Maven to generate code coverage reports.

1. Add JaCoCo Plugin in pom.xml
<build>
  <plugins>
    <plugin>
      <groupId>org.jacoco</groupId>
      <artifactId>jacoco-maven-plugin</artifactId>
      <version>0.8.8</version>
      <executions>
        <execution>
          <goals>
            <goal>prepare-agent</goal>
          </goals>
        </execution>
        <execution>
          <id>report</id>
          <phase>test</phase>
          <goals>
            <goal>report</goal>
          </goals>
        </execution>
      </executions>
    </plugin>
  </plugins>
</build>
2. Run Tests Using Maven
mvn clean test
This executes test cases and collects coverage data.
3. Generate the Report
mvn jacoco:report
Creates a detailed coverage report.
4. View the Report
Open the following file in a browser:
target/site/jacoco/index.html
5. What the Report Shows
Line coverage
Branch coverage
Method coverage
Class coverage
6. Optional: Enforce Coverage Rules

You can configure JaCoCo to fail the build if coverage is low.

Key Flow

👉 Add plugin → Run tests → Generate report → View HTML report

Define Docker and explain its key features. 
Docker

Docker is a containerization platform that allows developers to build, package, and run applications in containers.
Containers include the application along with all its dependencies, ensuring it runs consistently across different environments.

Key Features of Docker
1. Containerization
Packages application and dependencies into a single container.
Ensures consistency across development, testing, and production.
2. Portability
Docker containers can run on any system that supports Docker.
Works across different platforms and environments.
3. Lightweight
Containers share the host OS kernel.
Uses fewer resources compared to virtual machines.
4. Fast Deployment
Containers start quickly.
Enables rapid application deployment and scaling.
5. Isolation
Each container runs independently.
Prevents conflicts between applications.
6. Version Control
Docker images can be versioned.
Easy to track changes and roll back if needed.
7. Scalability
Supports scaling applications up or down easily.
Works well with orchestration tools like Kubernetes.
8. Reusability
Docker images can be reused across projects.
Saves development time.
9. Integration with DevOps
Supports CI/CD pipelines.
Automates build, test, and deployment processes.

It allows:
Uploading (pushing) images
Downloading (pulling) images
Access to public and private repositories

👉 Example: You can pull official images like nginx, mysql, etc.

describe the lifecycle of a Docker container. 
Lifecycle of a Docker Container

A Docker container goes through several states from creation to termination.

1. Created
Container is created from a Docker image.
Resources are allocated, but the container is not yet running.
2. Running
Container starts executing its application.
This is the active state where the container performs tasks.
3. Paused (Optional)
Container execution is temporarily suspended.
Processes are frozen without stopping the container.
4. Stopped
Container execution is stopped.
Data inside the container remains intact.
5. Restarted
A stopped container can be started again.
It resumes execution from its previous state.
6. Removed (Deleted)
Container is permanently deleted using commands like docker rm.
All data inside the container is lost (unless stored externally).
Lifecycle Flow

Created → Running → (Paused ↔ Running) → Stopped → Removed

differentiate between virtualization and containerization. 
Difference Between Virtualization and Containerization
1. Definition
Virtualization: Uses a hypervisor to create multiple virtual machines (VMs) on a single physical machine.
Containerization: Uses containers to run applications with shared OS resources.
2. Architecture
Virtualization: Each VM has its own OS (guest OS).
Containerization: Containers share the host OS kernel.
3. Resource Usage
Virtualization: Heavy, consumes more CPU, memory, and storage.
Containerization: Lightweight, uses fewer resources.
4. Startup Time
Virtualization: Slow (boots entire OS).
Containerization: Fast (starts application directly).
5. Performance
Virtualization: Slightly lower due to overhead.
Containerization: Better performance with minimal overhead.
6. Isolation
Virtualization: Strong isolation (separate OS for each VM).
Containerization: Process-level isolation (less strict than VMs).
7. Portability
Virtualization: Less portable due to full OS dependency.
Containerization: Highly portable across environments.
8. Examples
Virtualization: VMware, VirtualBox, KVM
Containerization: Docker
9. Use Case
Virtualization: Running multiple different OS environments.
Containerization: Deploying and scaling applications quickly.

how do you start, stop, and kill a Docker container?  
Starting, Stopping, and Killing a Docker Container
1. Start a Container
Starts an existing (stopped) container.
docker start <container_id_or_name>
2. Stop a Container
Gracefully stops a running container.
Allows the application to shut down properly.
docker stop <container_id_or_name>
3. Kill a Container
Forcefully stops a container immediately.
Does not allow graceful shutdown.
docker kill <container_id_or_name>
Key Difference
start → runs the container
stop → graceful shutdown
kill → immediate termination

what is the purpose of the volume parameter in a Docker run command? 
Purpose of the Volume Parameter in Docker Run Command

The volume parameter (-v or --volume) in the docker run command is used to mount a directory or file from the host system into a container.

Key Purposes
1. Data Persistence
Stores data outside the container.
Data is not lost even if the container is removed.
2. Sharing Data
Allows sharing data between:
Host and container
Multiple containers
3. Decoupling Data from Container
Separates application data from container lifecycle.
Makes containers reusable without losing data.
4. Easy Backup and Restore
Data stored on the host can be backed up easily.
5. Development Convenience
Developers can edit files on the host and see changes inside the container instantly.
Example
docker run -v /host/data:/container/data myimage
/host/data → directory on host
/container/data → directory inside container

What is virtualization, and what are the key components of a virtualization stack?  
Virtualization

Virtualization is a technology that allows multiple virtual machines (VMs) to run on a single physical machine by abstracting the underlying hardware.
Each VM runs its own operating system and applications independently.

Key Components of a Virtualization Stack
1. Physical Hardware
The actual machine (CPU, memory, storage, network).
Provides resources for virtualization.
2. Hypervisor (Virtual Machine Monitor)
Core software that enables virtualization.
Creates and manages virtual machines.

Types:

Type 1 (Bare-metal) → runs directly on hardware
Type 2 → runs on top of a host OS
3. Host Operating System
Present in Type 2 hypervisors.
Manages hardware and runs the hypervisor.
4. Guest Operating System
OS installed inside each virtual machine.
Runs applications independently.
5. Virtual Machines (VMs)
Software-based computers created by the hypervisor.
Each VM has its own OS, CPU, memory, and storage.
6. Virtual Resources
Virtualized components such as:
Virtual CPU
Virtual memory
Virtual disk
Virtual network interfaces
7. Management Tools
Tools used to monitor and control VMs.
Example: VM creation, allocation of resources, performance monitoring.

What is Puppet, and how do Puppet master and Puppet agents interact?  
Puppet

Puppet is a configuration management tool used to automate the deployment and management of system configurations across multiple servers.
It uses a declarative approach, where you define the desired system state, and Puppet ensures the system matches that state.

Interaction Between Puppet Master and Puppet Agents
1. Agent Initiates Request
The Puppet agent (client) runs on each node.
It periodically contacts the Puppet master (or manually triggered).
2. Authentication
The agent sends its SSL-encrypted credentials to the master.
The master verifies the agent.
3. Catalog Compilation
The Puppet master compiles a catalog based on:
Node configuration
Roles and policies
4. Catalog Sent to Agent
The compiled configuration (catalog) is sent to the agent.
5. Configuration Applied
The agent applies the configuration declaratively:
Installs packages
Configures services
Updates system settings
6. Periodic Updates
The agent checks in regularly (e.g., every ~30 minutes).
Ensures the system remains in the desired state.
Interaction Flow

Agent → Master (request) → Master compiles catalog → Agent applies configuration
